# Note to Windows Claude Code, from Linux Claude Code — raw capture data

Two raw `usbmon` text captures from Linux session 4, referenced throughout
`HANDOFF.md`'s "Linux session 4" section but not previously committed
(they only lived in a local scratch directory). Added here so the actual
evidence is available, not just prose summaries — matching the pattern
`usb_capture_session3`/`4` already established on your side.

Format note: these are Linux kernel `usbmon` text-interface captures
(`cat /sys/kernel/debug/usb/usbmon/5u`), **not** `.pcap`/`.pcapng` files —
a different capture backend than Windows' USBPcap, chosen because `tshark`
wasn't installed on this machine until partway through this session (now
fixed — see `HANDOFF.md`). One real limitation of this format worth
knowing: **the text interface truncates displayed payload data to the
first 32 bytes per line**, even though the numeric length field elsewhere
on the line (and the underlying transfer itself) is accurate — this cost
some initial confusion this session before being identified as a display
limitation, not a real truncation (cross-confirmed via `wLength` and every
test's own `Ok(N)` return value). If you're comparing byte-for-byte
against these files, be aware anything past byte 32 in a `0x0304` line
(51 bytes total) isn't shown.

## `usbmon_8zone_verify.txt`

Captured during `g615lr-8zone-batch.rs` (the 8-zone batched write test,
see `HANDOFF.md` "Linux session 4"). Used to directly verify the Rust
code's intended packet bytes actually match what goes out on the real USB
wire — they do, byte-for-byte, for everything the text format displays.
This is the capture behind the "packet construction fully exonerated"
conclusion.

## `usbmon_gui_direct.txt`

Captured live while a human manually clicked through modes in
`rog-control-center` (the Linux GUI), with `asusd` running normally (not
one of the raw test binaries — this is real, unmodified daemon/GUI
traffic). Two things this resolved:

1. **Confirms the GUI sends byte-identical `0x5d` traffic to the CLI** —
   same `b3`/`b5`/`b4`/`5a` four-packet sequence per mode change, no
   separate/different mechanism. Previously assumed (both go through the
   same D-Bus method) but now directly verified with real capture data.
2. **Explains a mid-session false alarm** where `Static`/`Breathe`/`Pulse`
   appeared to stop working via the GUI. Every `b3` packet in this capture
   has `colour1 = 00 00 00` — traced to a deliberate dark-baseline reset
   (`asusctl aura effect static -c 000000`, run right before this capture
   for a clean test baseline) whose cached colour got reused by the GUI's
   mode buttons. Not a code bug; confirmed fixed by setting real colours
   again. Full explanation in `HANDOFF.md` "Linux session 4."

## `usbmon_literal_12zone_replay.txt`

Captured during `g615lr-literal-12zone-replay.rs` — the most rigorous
`0x04` test this session: replays the LITERAL bytes (extracted via
`tshark` from `usb_capture_session4/multizone_12x_confirmed.pcapng`, not
regenerated by any Linux code) for all 16 zones, after real priming. This
is Windows' own exact captured bytes, sent byte-for-byte from Linux. All
16 `0x0304` writes are present in this capture (`grep -c 0304` → 16),
each returning success, matching what the program printed live
(`Ok(51)` for every write). **Visible result on the hardware: still just
RainbowCycle** — no different from any other test this session. This is
the citable evidence behind `HANDOFF.md`'s "packet content is now
exonerated as thoroughly as it is possible to exonerate it" conclusion.

Attempted to also capture this run as a proper `.pcapng` via live `tshark`
(matching Windows' format exactly, instead of the `usbmon` text format) —
didn't work. `tshark`/`dumpcap` refuses to open `/dev/usbmonN` even when
invoked through `sudo`; this is a known Wireshark hardening behavior
(`dumpcap` drops root privileges internally and requires genuine
`wireshark` group membership, which needs a fresh login/shell to take
effect after `usermod -aG wireshark`, not just `sudo` or `sg`). Not
pursued further this session since it's a tooling nicety, not core to the
finding — the `usbmon` text capture above is fully sufficient evidence on
its own. Worth revisiting with a fresh login if `tshark` live-capture
parity with Windows becomes valuable later.
